01

總覽與定位

從「單次 LLM 呼叫」到「多步驟編排」——LangChain、Agent 這類東西到底想解決什麼問題。

最單純的用法是：一段 prompt 進去、一段文字出來。但真實應用需要 LLM 做事——查今天的庫存、寫進資料庫、跨多個步驟完成一件任務。單次呼叫做不到這些：它不能執行動作、拿不到即時或私有資料、也無法可靠地把大任務拆成多步。

編排（Orchestration） 就是把 LLM 呼叫、工具、資料檢索與控制流組合成一個「能完成任務」的系統；Agent 則是編排的一種特殊形態——把「下一步做什麼」的決策權交給 LLM 自己。這也是 LangChain／LlamaIndex 等框架的共同命題。本篇刻意不綁定任何框架，只講底層心智模型，因為框架會換、原語不會。

LLM（推理核心）
reason / decide
負責理解、規劃、決定「下一步呼叫哪個工具、帶什麼參數」。它只輸出決策，不親自執行。
Tool（對外行動）
act / side-effect
真正與世界互動的手：查 API、讀資料庫、算數、送信。由你的 runtime 執行，不是模型。
編排層（控制流）
loop / route / chain
決定步驟怎麼串：固定 pipeline？條件分支？還是讓 LLM 動態決定（Agent）？
Context（狀態）
history / memory
承載對話歷史、工具結果、任務進度。每一步的推理都建立在當前 context 之上。
座標與 RAG、LLM Gateway 的分工

三者常被混談，但各管一段：RAG 解決「把對的資料塞進 context」（檢索）；LLM Gateway 解決「呼叫哪顆模型、怎麼控成本與流量」（模型路由）；本篇的編排／Agent 解決「把多次呼叫與工具串成一個能完成任務的流程」（控制流）。RAG 常常只是 Agent 手上的其中一個工具。

一句話答法編排是把 LLM、工具與控制流組成能完成任務的系統；Agent 是讓 LLM 自己決定下一步的編排形態。核心原語是 Tool Calling 與 Agent Loop。
02

核心原語：Tool / Function Calling

整個 Agent 世界的最小積木——讓模型「決定呼叫哪個函式、帶什麼參數」的機制。

Function Calling（也叫 Tool Calling）是這樣運作的：你把可用工具的「規格」告訴模型，模型在推理後不是回一段自然語言，而是回一個結構化的呼叫意圖（工具名稱＋參數）。它自己不執行，執行是你的 runtime 的責任。四個環節：

STEP 1
宣告工具
開發者把工具的 name／description／參數 schema（通常是 JSON Schema）連同 prompt 一起送給模型。
STEP 2
模型決定呼叫
模型判斷「這題需要工具」，輸出結構化 tool call：name + arguments，而非散文。
STEP 3
Runtime 執行
你的程式攔截這個意圖，真正去呼叫 API／查 DB／算數，拿到結果。模型全程沒碰到真實系統。
STEP 4
結果回寫
把工具結果放回 context，模型據此續推理——回答使用者，或決定再呼叫下一個工具。

一個工具規格長這樣（框架無關，概念上就是「給模型看的函式簽章」）：

{
  "name": "get_order",
  "description": "用訂單編號查詢訂單狀態與物流單號。當使用者詢問訂單進度時使用。",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": { "type": "string", "description": "訂單編號，如 A1234" }
    },
    "required": ["order_id"]
  }
}

模型看得到的只有這份「說明書」。它靠 description 判斷何時該用、靠 parameters 知道要湊哪些參數——所以描述寫得爛，模型就召喚錯工具或亂填參數。

關鍵模型只「決定」，不「執行」

這是最容易誤解的一點：LLM 輸出的 tool call 只是一個意圖（intent），像填好一張申請單。真正去執行、去碰資料庫與外部 API 的，是你的 runtime。這個「決策」與「執行」的分離，正是後續 Agent Runtime 後端服務設計 與 Tool Gateway 安全邊界 兩篇要深入的地方——因為執行點就是風險點。

schema 欄位作用寫不好會怎樣
name 工具識別 命名含糊（do_stuff）→ 模型難判斷用途
description 何時該用 最關鍵：描述不清 → 該用時不用、不該用時亂用
parameters 參數約束 缺型別／必填約束 → 模型湊出無效參數，執行時炸
回傳（慣例） 結果格式 結構混亂 → 模型讀不懂結果，下一步推理走偏

工具規格是「人給模型的介面契約」；契約寫得好壞，直接決定 Agent 可靠度。

陷阱工具不是越多越好

塞幾十個工具給模型，反而讓它選擇困難、召喚錯工具、還吃掉大量 prompt token。實務上常用「工具分組／依情境只暴露相關子集」來收斂。這也預示了 MCP 這類協定要解的問題——工具太多、太雜、來源太散時，需要標準化的發現與描述方式（見 s6）。

一句話答法Function Calling 讓模型輸出「呼叫哪個工具＋參數」的結構化意圖，真正執行由 runtime 負責；工具的 description 品質決定模型會不會、正不正確地呼叫。
03

Agent Loop：推理 → 行動 → 觀察

把「單次 tool call」升級成「持續迴圈」——這就是一個 Agent 的心臟。

單次工具呼叫只能處理「一步就能答」的問題。真正的 Agent 把 Tool Calling 包進一個迴圈：模型推理（Thought）下一步 → 行動（Action，發出 tool call） → runtime 執行後回傳觀察（Observation） → 模型讀完觀察再推理下一步……如此反覆，直到它認為資訊足夠、輸出最終答案。這個「Reason + Act」的迴圈常被稱為 ReAct 模式。

Thought 推理 → Action 呼叫工具 → Observation 觀察結果 ↺ 尚未收斂就回到 Thought → Final Answer

這個迴圈每一輪都攜帶並更新一些狀態：

  • 對話／執行歷史：使用者問題 + 歷次 Thought／Action／Observation，全部累積在 context 裡，一起餵回模型。
  • 可用工具集：這一輪模型能選用的工具。
  • 步數計數：走了幾輪——這是護欄的錨點（見下）。
陷阱迴圈為什麼會失控？——一定要有終止護欄

迴圈的終止條件是「模型輸出不含 tool call 的最終答案」。但模型可能永遠不收斂：反覆呼叫同一個工具、在兩個動作間鬼打牆、或因為某步觀察看不懂而一直重試。因此每個正式 Agent 都必須設硬護欄：最大步數（max iterations）、逾時、以及重複動作偵測。撞到上限就中止並回報，而不是無限燒 token。

關鍵context 隨迴圈膨脹——這是成本與延遲的根源

因為每一輪都把「歷史全文」再送一次給模型，走得越多步、context 越長，每一步的 token 成本與延遲都在往上疊。一個 5 步的 Agent，最後一步吃的 token 可能是第一步的好幾倍。這解釋了為什麼 Agent「能做很多事」的同時「又慢又貴」——s5 會把這筆帳算清楚。

一句話答法Agent Loop＝Thought→Action→Observation 反覆到收斂；狀態是不斷累積的 context，終止靠模型自己停或撞上 max-iterations 護欄。
04

編排模式光譜：Workflow vs Agent

不是「有沒有用 Agent」的二分法，而是一條從「確定性」滑到「自主性」的光譜。

業界一個好用的框架（框架無關）是把編排分成兩端：Workflow 是「LLM 與工具照預先寫死的程式路徑跑」；Agent 是「LLM 動態決定自己的流程與工具用法」。在這兩端之間有一整排常見模式，但要注意——除了最後的 Autonomous Agent，其餘全都還是 Workflow 的變體，差別只在 LLM 在既定結構內被賦予多少裁量權：

← 確定性 / 可預測 / 好除錯 自主性 / 靈活 / 難預測 →
模式控制流何時用
Prompt Chaining 固定序列 任務能乾淨拆成穩定步驟（草稿→翻譯→潤稿）
Routing 先分類再分流 輸入類型多樣，各走不同專用處理（客服工單分流）
Parallelization 並行 + 匯總 可切分工作或需多次投票取共識（多面向審查）
Orchestrator–Workers 協調者動態拆派 子任務數量／內容事前未知，由 LLM 現場拆解
Evaluator–Optimizer 生成→評估→改寫迴圈 有明確評分標準、值得多輪打磨（翻譯校準）
Autonomous Agent LLM 全程自主決策 路徑無法預先枚舉、需長尾應變（開放式任務）

關鍵：除最後一項外，其餘五種都屬 Workflow（結構由工程師預先定義）；由上到下遞增的是「LLM 在既定結構內的裁量權」，只有 Autonomous Agent 才真正跨進 Agent。

關鍵預設從左邊開始

Agent 不是越自主越高級。自主性換來的是不可預測、難除錯、成本浮動。實務準則是：能用固定 Workflow 解決的，就別上 Agent；只有當「路徑真的無法事先枚舉」時，自主性的代價才划算。下面的互動 demo 用同一個任務，讓你直接看到這個取捨。

任務：使用者問「我的訂單 #A1234 什麼時候會到？」左軌是寫死的 pipeline，右軌是自主決策的 Agent。同一題，看兩者路徑怎麼分岔。

一句話答法Workflow＝路徑寫死、可預測；Agent＝LLM 動態決策、能應變長尾但不可預測。預設用 Workflow，路徑無法事先枚舉時才付出自主性的代價。
05

工程挑戰：可靠性 · 成本 · 可觀測性

Demo 裡「Agent 靈活勝出」很美好，但正式上線時，這三件事會反咬你一口。

把 Agent 從 demo 推到生產環境，難的從來不是「讓它動起來」，而是「讓它穩定、可負擔、看得見」。三個維度各有典型陷阱：

維度典型問題常見緩解
可靠性 幻覺 tool call、無限迴圈、工具回錯不會處理 max iterations 護欄、參數校驗、工具錯誤回饋給模型重試、關鍵動作要人審
成本 context 逐輪膨脹、每步都重送全歷史、步數失控 限制步數與工具數、歷史裁剪／摘要、快取、小任務用小模型
延遲 工具串行、每輪一次模型往返，累加起來很慢 可並行的工具並行、串流回覆、把確定步驟抽回 Workflow
可觀測性 多步決策是黑盒，出錯難重現、難歸因 完整 trace 每一步 Thought／Action／Observation、記錄 token 與耗時、可重播

Agent 的三筆帳：對不對（可靠性）、貴不貴（成本／延遲）、看不看得見（可觀測性）。

陷阱幻覺 tool call：連「呼叫哪個工具」都可能是假的

模型可能召喚根本不存在的工具、或給存在的工具湊出無效參數。因為它輸出的只是「看起來合理的文字」，不保證對應真實系統。所以 runtime 必須把 tool call 當不可信輸入校驗：工具名在不在白名單？參數過不過 schema？——這正好接上 Tool Gateway、權限與安全邊界 那篇的核心。

關鍵人在迴圈裡（Human-in-the-loop）

對「不可逆或高風險」的動作（退款、刪資料、寄信給客戶），成熟做法是讓 Agent 走到那一步時暫停、等人確認再執行。把自主性限制在「安全的動作」上，危險動作留一道人為閘門——這是自主性與可控性之間最實用的折衷。

一句話答法Agent 上線的三道坎是可靠性（幻覺／迴圈，靠護欄與校驗）、成本延遲（context 膨脹，靠限步與裁剪）、可觀測性（黑盒，靠全程 trace）；高風險動作留人審。
06

為何需要標準協定（MCP / A2A / ACP）

當工具、資料源、甚至別的 Agent 越來越多，「每對整合都自己刻」會爆炸——這就是協定要解的問題。

前面每個工具都要你手寫 schema、手接執行邏輯。當你有 M 個 Agent／應用、要接 N 個工具或資料源時，若每一對都各自客製，整合成本會逼近 M × N——每加一個新工具，所有 Agent 都要重接一次。這正是幾套新興協定想解的「整合爆炸」問題：

協定連接的是解決的核心問題
MCP
Model Context Protocol		 模型 ↔ 工具 / 資料 用標準協定描述與發現工具、資源、prompt，讓「一次實作、到處可用」，取代 M×N 客製
A2A
Agent-to-Agent		 Agent ↔ Agent 讓不同團隊、不同框架做的 Agent 能互相委派任務與溝通
ACP
Agent Comm. Protocol		 Agent 互通 / 對照 與 A2A 同屬「Agent 間互通」路線，定位與取捨各有差異

把「工具接法」與「Agent 溝通」標準化，就像 USB／HTTP 之於硬體與網路——重點是通用介面，不是各自為政。

座標本篇只給動機，細節在後續三篇

這裡刻意只講「為什麼需要標準協定」。角色分工（Host／Client／Server）、傳輸方式（stdio／HTTP+SSE）、訊息格式、與自建 Function Calling 的差異等，屬於 Model Context Protocol 深度解析、A2A Protocol、ACP 與 Agent 互通 三篇的守備範圍。你現在只需要記住那個動機：M×N 客製整合不可持續，所以要標準化。

一句話答法Agent × 工具的整合會走向 M×N 爆炸；MCP 標準化「模型↔工具」、A2A／ACP 標準化「Agent↔Agent」，讓整合從客製變成插拔。細節留給協定專篇。
07

面試折衷與實務選型

最值錢的判斷力不是「會不會搭 Agent」，而是「知道什麼時候不該用 Agent」。

面對一個需求，先問自己：這任務的路徑能不能事先枚舉？能，就用越確定性的方案越好；不能，才逐步往自主端移動。一個實用的決策階梯：

如果……就用理由
單次呼叫就能答，不需外部資料／動作 純 Prompt 最便宜、最可預測，別過度設計
需要外部資料或動作，但步驟固定 Workflow（Chaining／Routing） 可預測、好測試、成本可估，涵蓋大多數場景
路徑無法事先枚舉、需長尾應變 Agent（自主迴圈） 用可預測性換靈活性；務必配護欄與可觀測性
含不可逆／高風險動作 Agent + Human-in-the-loop 自主歸自主，危險動作留人為閘門

選型階梯：從左（便宜可控）往右（靈活昂貴）逐級升，能停在左邊就別往右。

面試高頻追問怎麼答

Q：什麼時候該用 Agent、什麼時候用 Workflow？ 答：路徑能事先枚舉就用 Workflow（可預測、好除錯、成本可估）；只有路徑無法枚舉、需長尾應變時才上 Agent，並付出不可預測與成本浮動的代價。
Q：Agent 上線最大的風險是什麼？ 答：可靠性（幻覺 tool call、無限迴圈）、成本（context 逐輪膨脹）、可觀測性（多步黑盒）。對應要有 max-iterations 護欄、參數校驗、歷史裁剪、全程 trace。
Q：為什麼需要 MCP 這類協定？ 答：Agent×工具的整合會走向 M×N 客製爆炸，標準協定讓工具「一次實作、到處插拔」。

一句話答法能用 Prompt 就別上 Workflow，能用 Workflow 就別上 Agent；自主性是拿「可預測、可控、成本穩定」換來的，只在路徑無法枚舉時才划算，且高風險動作永遠留人審。
