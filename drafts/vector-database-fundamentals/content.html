01

總覽與定位

向量資料庫解決什麼痛？與全文檢索、OLTP 在 RAG / 語意搜尋版圖中的座標。

當使用者問「有沒有跟這段描述很像的文件？」或「找一張長得像這張貓的照片」，傳統 B-Tree 或倒排索引無法比對語意相似度——你需要把文字 / 圖片先轉成高維向量（Embedding），再用近似最近鄰（ANN, Approximate Nearest Neighbor）在毫秒內從數百萬～數十億筆向量中找出 Top-K 最相近的結果。

向量資料庫（Vector Database）的核心使命，是以 ANN 索引結構承擔高維相似度搜尋，常見於 RAG 知識庫檢索、推薦系統、以圖搜圖、異常偵測等場景。它與上一篇的全文檢索引擎（Elasticsearch）互補：ES 擅長 keyword / BM25 精確匹配，向量搜尋擅長語意近似——現代 AI 應用常兩者並用（Hybrid Search）。

座標Vector DB vs Search Engine vs OLTP

• OLTP：point lookup、transaction，不擅長高維 cosine / L2 相似度掃描。
• Search Engine（ES）：倒排索引 + BM25，keyword / 全文匹配，可疊加 dense_vector 但非原生強項。
• Vector DB（Milvus / pgvector）：ANN 索引（HNSW / IVF）專為高維向量 Top-K 優化，recall 與 latency 可調。
實務常見：PostgreSQL (metadata) + pgvector (小規模) → Milvus (億級向量)，或 ES hybrid = BM25 + kNN。

維度 OLTP Search Engine Vector DB
索引結構 B-Tree / Hash Inverted Index HNSW / IVF / DiskANN
典型查詢 WHERE id = ? match "關鍵字" Top-K ANN(query_vec)
相似度度量 等值比對 BM25 / TF-IDF Cosine / L2 / Inner Product
延遲目標 毫秒級 point lookup 毫秒～百毫秒 毫秒～十毫秒（視 ANN 參數）

面試高分：能說清楚「為什麼不能 brute-force 掃全表」以及向量搜尋在 RAG pipeline 中的位置。

一句話答法向量 DB 以 ANN 索引承擔高維語意相似度 Top-K 搜尋，是 RAG 檢索層的核心；與 ES 的 keyword 搜尋互補而非取代。
02

Embedding 管線：從原始資料到可搜尋向量

重點在「向量如何進索引、如何被查詢」，而非選哪個 embedding 模型。

向量搜尋的資料流可拆成四步：Chunk → Embed → Index → Query Embed → ANN Retrieve。System Design 面試中，考官更關心索引與檢索折衷，而非模型細節——但以下管線概念必須能畫出來。

  1. Chunking：長文件切分成固定長度片段（如 512 tokens），每 chunk 獨立 embedding。Chunk 策略影響 recall——切太細語意破碎，切太粗檢索粒度太粗。
  2. Embedding：呼叫 embedding API（OpenAI text-embedding-3、Cohere、開源 BGE 等）將 chunk 轉成 d 維浮點向量（常見 d=768～1536）。Index 與 Query 必須用同一模型、同一維度。
  3. Metadata 附帶：向量通常與 doc_id、source、timestamp、tenant_id 等 scalar 欄位一起儲存，供 pre-filter（先過濾再 ANN）或 post-filter（ANN 後再過濾）。
  4. Query Path：使用者問題同樣 embed 成 query vector，送入 ANN 索引取 Top-K，再將 chunk 文本餵給 LLM 做 RAG 生成。
白話為什麼維度與模型不能混用？

768 維 OpenAI embedding 與 1024 維 BGE 向量不在同一語意空間——混用會導致相似度毫無意義。Production 變更 embedding 模型 = 全量 re-embed + 重建 ANN 索引，需規劃 alias 切換與雙寫過渡期。

管線階段 常見痛點 設計對策
Chunking 表格 / 程式碼被切斷 結構感知切分、重疊窗口（overlap 10～20%）
Embed 吞吐 億級文件 embed 慢 批次 API、GPU worker pool、增量 CDC embed
Index 更新 新 doc 何時可搜 近即時 upsert vs 批次 rebuild（類 ES refresh）
Query 語意對但 keyword 漏掉 Hybrid：BM25 + vector fusion（RRF）

本手冊不深入 embedding 模型選型；面試聚焦「管線怎麼接、索引怎麼維護」。

陷阱以為「embed 完就完事」

向量品質取決於 chunk 品質、metadata filter 設計、以及 ANN 的 recall@K。RAG 幻覺常來自檢索漏掉正確 chunk（低 recall），而非 LLM 本身——調 ANN 參數與 hybrid 往往比換更大模型更有效。

一句話答法Embedding 管線 = Chunk → Embed → Index → Query Embed → ANN；index/query 必須同模型同維度，變更模型需全量 re-embed + 重建索引。
03

ANN 索引：HNSW、IVF 與 Recall / Latency 取捨

Brute-force 精確 kNN 在億級向量不可行——ANN 用近似換速度，參數決定 recall 與延遲。

給定 query vector q 與資料集 N 個 d 維向量，精確 kNN 需計算 N 次距離——O(N·d)，億級向量單次查詢需數秒。ANN 索引透過資料結構剪枝，將複雜度降到 sub-linear，以略降 recall換毫秒級延遲。

主流 ANN 索引

索引 原理 建索引成本 查詢特性
Flat / Brute-force 全表掃描，100% recall 低（無索引結構） 小資料量基準線；大資料不可行
HNSW 分層 navigable small world 圖，貪婪圖遍歷 高（記憶體密集） 高 recall、低 latency；Milvus / pgvector 預設首選
IVF (Inverted File) K-means 分 nlist 個 cluster，查詢只搜 nprobe 個 中 nprobe↑ recall↑ latency↑；億級常 IVF + PQ 壓記憶體
DiskANN 圖索引存 SSD，減少 RAM 壓力 高 十億級向量、記憶體受限場景

HNSW 參數 M（每節點邊數）、efConstruction（建索引寬度）影響索引品質；查詢時 ef 越大 recall 越高、延遲越高。

相似度度量

  • Cosine Similarity：方向相似，text embedding 最常用；向量常先 L2 normalize。
  • L2 (Euclidean) Distance：絕對距離，圖像 embedding 常見。
  • Inner Product (IP)：normalize 後等價 cosine；未 normalize 時有量級差異。
關鍵Recall@K vs Latency 曲線

Production 需用離線 benchmark畫 recall-latency 曲線：固定 query set，掃描 ef / nprobe 參數，找到業務可接受 recall（如 95%）對應的 P99 latency。面試常問：「如何在不換硬體下提升 recall？」→ 調高 ef、增大 M、或 hybrid 補漏。

陷阱只看 latency 不看 recall

把 ef=16 壓到 1ms 很炫，但 recall@10 可能只有 70%——RAG 漏掉 30% 正確 chunk 直接傷害回答品質。SLA 應定義recall@K ≥ X% 且 P99 ≤ Y ms，兩者缺一不可。

一句話答法ANN 用 HNSW / IVF 等結構剪枝換速度；HNSW 的 ef/M 參數控制 recall-latency 曲線，Production 需離線 benchmark 定 SLA。
04

pgvector vs Milvus：何時先別急著上專用 DB？

PostgreSQL 擴充 pgvector 夠用時不必引入新基礎設施；億級向量才需要原生 Vector DB 的分散式 ANN。

pgvector：PostgreSQL 上的向量擴充

pgvector 是 PostgreSQL extension，在既有 PG 表加 vector(d) 欄位，支援 HNSW / IVFFlat 索引與 SQL 混合查詢（metadata filter + ANN 一條 SQL 搞定）。

  • 優勢：零新基礎設施、ACID transaction、JOIN metadata、團隊已會 PG、百萬級向量足夠。
  • 限制：單機 PG 記憶體 / CPU 天花板；HNSW 全在 RAM，十億向量撐不住；寫入吞吐受 PG WAL 限制；無原生 shard 水平擴展。
  • 典型場景：MVP RAG、< 500 萬 chunk、metadata 複雜 filter、小團隊快速上線。
-- pgvector 典型 DDL
CREATE EXTENSION vector;
CREATE TABLE doc_chunks (
  id BIGSERIAL PRIMARY KEY,
  doc_id UUID NOT NULL,
  content TEXT,
  embedding vector(1536),
  tenant_id INT
);
CREATE INDEX ON doc_chunks
  USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);

-- 混合 filter + ANN
SELECT id, content, embedding <=> $1 AS dist
FROM doc_chunks
WHERE tenant_id = 42
ORDER BY embedding <=> $1
LIMIT 10;

Milvus：原生向量資料庫

Milvus 是專為向量搜尋設計的分散式 DB，Collection → Partition → Segment 層次，內建 HNSW / IVF_FLAT / IVF_PQ / DiskANN，支援水平 shard、副本、專用 query node。

  • 優勢：億～十億向量、寫入 / 查詢水平擴展、多種索引與量化（PQ 壓記憶體）、雲託管（Zilliz Cloud）。
  • 限制：額外運維成本、metadata 不如 PG 靈活（需 scalar field + expression filter）、 eventual consistency 語意需理解。
  • 典型場景：電商以圖搜圖、大規模 RAG、推薦召回、多 tenant 億級向量。
決策維度 pgvector Milvus
向量規模 < 500 萬 chunk 舒適 億級+ 原生設計
Metadata 查詢 SQL JOIN / WHERE 最強 Scalar filter expression
Transaction PG ACID 無跨 collection ACID
運維成本 已有 PG 則極低 需專用叢集 / 託管
索引選擇 HNSW / IVFFlat HNSW / IVF / PQ / DiskANN
遷移路徑 MVP 起點 pgvector 瓶頸後遷移

面試高分：「先用 pgvector 驗證產品，QPS / 向量量 / recall SLA 觸頂再遷 Milvus」——展現漸進式架構而非過早優化。

實務遷移 pgvector → Milvus 的 SOP

1. 雙寫：新 chunk 同時寫 PG + Milvus（CDC 或 application dual-write）。
2. 離線 benchmark：同一 query set 比對 recall@K 與 P99 latency。
3. 讀流量灰度：10% → 50% → 100% 切換 ANN 查詢到 Milvus。
4. PG 保留 metadata source of truth，Milvus 專職向量檢索。

陷阱Day 1 就上 Milvus

10 萬 chunk 的內部知識庫引入 Milvus 叢集——運維、監控、版本升級成本遠超收益。先用 pgvector 或託管向量服務（Pinecone / Zilliz Serverless）驗證，量級到了再自建。

一句話答法pgvector 適合百萬級、已有 PG、metadata 複雜的 MVP；Milvus 適合億級向量、需水平擴展與多種 ANN 索引的 Production 規模。

06

面試考點與架構折衷

RAG 檢索層怎麼設計？Hybrid Search 怎麼做？什麼時候該從 pgvector 遷到 Milvus？

RAG 檢索層架構

  • Ingestion：文件 → chunk → embed → upsert vector DB（+ metadata）。
  • Retrieval：query embed → ANN Top-K →（optional）reranker 精排 → 餵 LLM context。
  • Hybrid：BM25（ES）+ Vector（Milvus）各取 Top-K，用 RRF（Reciprocal Rank Fusion） 合併——keyword 精確 + 語意近似互補。
  • Pre-filter vs Post-filter：tenant / ACL filter 在 ANN 前（效率高但索引需支援）或 ANN 後（簡單但 recall 可能漏）。
考點 1「RAG 檢索效果不好，怎麼排查？」

回答框架：
1. 量 recall@K：用標注 query-chunk pair 離線評估。
2. 低 recall → 調 ANN ef/nprobe、檢查 chunk 策略、試 hybrid BM25+vector。
3. 高 recall 但 LLM 仍錯 → reranker（cross-encoder）或 prompt 問題。
4. 延遲高 → 減 K、用 PQ 量化、或 query node 水平擴展。

考點 2「pgvector 夠用嗎？什麼時候換 Milvus？」

• 向量 < 500 萬、QPS < 數百、metadata 需 SQL JOIN → pgvector 通常夠。
• 觸頂信號：PG CPU / 記憶體飽和、HNSW 建索引超時、P99 > SLA、需跨機 shard。
• 遷移策略：雙寫 + 灰度讀 + PG 保留 metadata SoT。

陷阱忽略向量索引的更新成本

HNSW 不擅長頻繁刪改——大量 delete 後圖結構退化，需 periodic rebuild 或改用 IVF（對寫入友善但 recall 略低）。Milvus 的 segment compaction 與 pgvector 的 REINDEX 都需納入 SOP。

Trade-off 選 A 選 B 決策線索
儲存 pgvector (PG) Milvus 專用 百萬級 MVP vs 億級 Production
索引 HNSW 高 recall IVF+PQ 省記憶體 RAM 充足 vs 十億向量
搜尋模式 Pure vector ANN Hybrid BM25+vector 專有名詞多 → hybrid；純語意 → vector
Filter Pre-filter Post-filter filter 選擇性高 → pre-filter
SLA recall@10 ≥ 95% P99 ≤ 50ms 兩者都要，用 benchmark 找平衡點

向量 DB + ES hybrid + LLM 三層是現代 RAG 應用的標準檢索生成分層。

一句話答法向量 DB 是 RAG 檢索核心：pgvector 做 MVP、Milvus 做億級；ANN 參數定 recall-latency SLA，檢索差先調索引與 hybrid 再怪 LLM。
