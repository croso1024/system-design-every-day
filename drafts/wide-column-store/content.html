01

總覽與定位

Wide-column Store 是什麼、解決什麼痛、與關聯式 / KV / Document DB 在系統設計版圖中的座標。

寬列儲存（Wide-column Store）是一種以稀疏、可變欄位組織資料的 NoSQL 模型：每列由 Row Key 唯一識別，底下可掛多個 Column Family，每個 family 內又有動態的 column qualifier 與 timestamp 版本。它不像關聯式用固定 schema 的 table，也不像純 KV 只存 blob——而是在單一 partition 內支援範圍掃描與多版本，專為海量寫入、水平擴展、時間序列與事件流而生。

面試與實務上，Cassandra 是最常被問的 wide-column 代表（masterless ring、最終一致、CQL）；HBase 則是 Hadoop 生態的對照組（HMaster + RegionServer、強一致讀寫路徑）。兩者共享「partition + 範圍排序」的思維，但一致性模型與運維拓樸截然不同。

座標Wide-column vs 其他儲存模型

• 關聯式（PostgreSQL）：JOIN、正規化、任意 ad-hoc 查詢——靈活但垂直擴展有上限。
• Document（MongoDB）：JSON 文件、靈活 schema，單文件讀寫強，跨文件 JOIN 弱。
• KV（Redis / DynamoDB 純 KV 模式）：極簡 key → value，延遲低，複雜查詢需應用層拼裝。
• Wide-column（Cassandra / HBase）：依 partition key 水平切分、clustering key 範圍排序，為已知查詢模式優化，換取 PB 級寫入吞吐。

維度 Cassandra HBase PostgreSQL
拓樸 Masterless P2P ring HMaster + RegionServer Primary + Replica
一致性 可調（預設最終一致） 單 Region 內強一致 強一致（單主）
查詢語言 CQL（類 SQL） API / Phoenix SQL 完整 SQL
設計思維 依查詢設計 table 依 Row Key 設計 依實體正規化
典型場景 IoT 時序、訊息收件箱、產品目錄 Hadoop 批次分析、離線特徵庫 交易 SoT、複雜報表

面試高分：能說出 wide-column 不是「沒 schema 的 SQL」，而是用 partition key 換取寫入擴展，查詢必須帶 partition key 才高效。

一句話答法Wide-column Store 用 Row Key 切 partition、clustering column 做範圍排序，專為海量寫入與已知查詢模式設計；Cassandra 走 masterless 最終一致，HBase 走 master-based 強一致，兩者都比關聯式更強調「先想查詢再建模」。
02

核心資料模型：Partition Key 與 Clustering Key

Cassandra CQL 的 PRIMARY KEY 語意、稀疏欄位、以及一筆 row 在磁碟上如何被定位。

在 Cassandra 的 CQL 裡，PRIMARY KEY 決定資料如何分片與排序，這是 wide-column 與關聯式最大的語意差異：

  • Partition Key：決定這列落在哪個節點（透過 consistent hashing）。同一 partition key 的所有列必定在同一 replica set。
  • Clustering Key：在 partition 內決定排序與範圍掃描順序，支援 WHERE pk = ? AND ck > ? 這類區間查詢。
  • Static columns：屬於整個 partition 的共用欄位（例如某使用者的 display name），不隨 clustering row 重複。
-- 訊息收件箱：依 user_id 分 partition，依 message_id 排序
CREATE TABLE inbox_by_user (
  user_id       UUID,
  message_id    TIMEUUID,
  sender        TEXT,
  subject       TEXT,
  body          TEXT,
  PRIMARY KEY (user_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);

-- 查詢必須帶 partition key
SELECT * FROM inbox_by_user
WHERE user_id = ? AND message_id > ?
LIMIT 50;

HBase 用不同術語表達同一概念：Row Key 同時承擔 partition 與排序職責（可人工設計 salt / reverse timestamp），Column Family 是物理儲存單元，Column Qualifier 是動態欄位名，每個 cell 帶 timestamp 版本。

陷阱把 Cassandra 當 PostgreSQL 用

沒有 partition key 的 WHERE 會觸發全叢集 scatter-gather，每個節點掃自己的資料再合併——延遲與負載隨節點數線性成長。ALLOW FILTERING 不是救命稻草，是生產環境的紅燈。設計時先列出Top 5 查詢，再反推 table schema。

關鍵稀疏性與多版本

同一 table 中，row A 可能有 3 個 column，row B 有 50 個——schema 不需預先宣告所有欄位。HBase 每個 cell 可存多個 timestamp 版本，適合「只保留最近 N 版」的時序或變更追蹤；Cassandra 透過 TTL 與 USING TIMESTAMP 達成類似效果。

一句話答法Partition key 決定資料落在哪台機器；clustering key 決定 partition 內怎麼排與怎麼掃——沒帶 partition key 的查詢在 wide-column 世界裡等同全表掃描。
03

Cassandra 架構：Masterless Ring 與最終一致

Gossip、Replication Factor、Quorum 讀寫、以及為何能無單點寫入擴展。

Cassandra 採用對等節點 ring，沒有 central master。每個節點透過 Gossip 交換 cluster 狀態（誰活著、token 範圍、schema 版本），client 可用 any node as coordinator 發起請求。

寫入路徑（簡化）

  1. Client 送寫入到任一 coordinator。
  2. Coordinator 依 partition key 算出 token，找出負責的 replica nodes（由 Replication Factor RF 決定副本數）。
  3. Coordinator 將寫入並行推送到 replicas；依 Consistency Level 等待足夠節點回應（例如 QUORUM = 過半副本確認）。
  4. 背景 hinted handoff / anti-entropy repair 修復暫時不一致的副本。
W + R > RF → 強一致讀寫（在無失敗節點時）

其中 W 為寫入一致性所需確認數、R 為讀取一致性所需確認數、RF 為副本數。預設常見 RF=3、ONE 讀寫——吞吐高但可能讀到舊值；金融級場景才調高到 QUORUM / LOCAL_QUORUM。

Consistency Level 行為 適用
ONE 任一副本回應即成功 高吞吐寫入、可容忍短暫 stale read
QUORUM 過半副本確認 需要較新資料、跨 DC 需注意延遲
LOCAL_QUORUM 本地 DC 內過半副本 多 DC 部署，避免跨洋 quorum 延遲
ALL 全部副本確認 極少使用；任一節點掛掉即寫入失敗
實務多 DC 與 NetworkTopologyStrategy

生產環境用 NetworkTopologyStrategy 把副本分散到不同 rack / DC，搭配 LOCAL_QUORUM 讓讀寫在本地 DC 完成，跨 DC 靠 async replication 與 repair 追平。這是 Cassandra 在全域部署時的標準打法。

一句話答法Cassandra 無 master，coordinator 依 token 找 replica 並行寫入；一致性由 CL 與 RF 共同決定，預設偏可用性與吞吐，需主動調高 quorum 才接近強一致。
04

HBase 對照：Master-based 與強一致路徑

HMaster、Region、WAL、MemStore——以及與 Cassandra 在一致性與運維上的取捨。

HBase 建立在 HDFS 之上，架構明確分層：

  • HMaster：管理 Region 分配、schema 變更、failover（不是資料路徑上的單點，但 metadata 操作依賴它）。
  • RegionServer：實際服務讀寫，每個 Region 是連續 row key 範圍的 shard。
  • ZooKeeper：協調 master 選舉與 region 狀態。

寫入路徑

寫入先 append 到 WAL（Write-Ahead Log），再進 MemStore；MemStore 滿了 flush 成 HDFS 上的 HFile。讀取需合併 MemStore + 多個 HFile（類 LSM-Tree），背景 Compaction 合併小檔。單 Region 內讀寫走同一 RegionServer，語意上接近強一致。

對比維度 Cassandra HBase
控制平面 無中心 master（Gossip） HMaster + ZK
儲存底層 本地 SSTable（自建） HDFS HFile
預設一致性 最終一致（可調） 單 Region 強一致
最佳場景 線上服務、低延遲寫入 與 Hadoop 批次共存的離線/近線
擴展痛點 hot partition、repair 成本 Region split、Major compaction IO
陷阱以為 HBase 適合低延遲 OLTP

HBase 為高吞吐批次寫入 + 大範圍 scan 優化，單筆隨機讀寫延遲通常高於 Cassandra。若你要毫秒級線上 API 且無 Hadoop 依賴，Cassandra / Scylla 更常見；HBase 適合「寫入特徵庫、MapReduce 讀、離線訓練」這類路徑。

一句話答法HBase 是 master-based + HDFS 的強一致 wide-column，擅長與 Hadoop 批次共存；Cassandra 是 masterless 最終一致，擅長線上服務的水平寫入擴展——選誰看你是 OLTP 還是離線分析。
05

依查詢設計資料模型

從 access pattern 反推 schema、反正規化、多表物化——與關聯式「先正規化再優化」的思維翻轉。

關聯式教科書教的是：消除冗餘 → 正規化 → 需要時再 denormalize。Wide-column 幾乎反過來：先列出所有讀寫查詢，再為每條查詢設計一張（或一種）table。Cassandra 不支援跨 partition JOIN，Document DB 的 $lookup 在這裡也不存在。

實戰範例：電商訂單

需求：

  • Q1：依 user_id 查最近訂單列表
  • Q2：依 order_id 查單筆訂單詳情
  • Q3：依 product_id 查某商品的近期訂單（營運報表）

關聯式一張 orders table + 索引搞定；Cassandra 需要至少兩張物化視角：

-- Q1：收件箱模式
CREATE TABLE orders_by_user (
  user_id UUID, order_id TIMEUUID, total DECIMAL, status TEXT,
  PRIMARY KEY (user_id, order_id)
) WITH CLUSTERING ORDER BY (order_id DESC);

-- Q2：直接 lookup（order_id 作 partition key）
CREATE TABLE orders_by_id (
  order_id UUID PRIMARY KEY,
  user_id UUID, total DECIMAL, status TEXT, items TEXT
);

-- Q3 若流量大，再建 orders_by_product；否則用批次工具從 Q2 匯出
關鍵寫入放大是可接受的 Trade-off

建立訂單時可能需同時寫入多張 table（dual write 或 async fan-out）。在關聯式這是冗餘；在 Cassandra 這是為讀取效能付的設計代價。面試要主動講出這個 trade-off，並提到可用 Kafka / CDC 做 eventual 同步降低雙寫風險。

避免 Hot Partition

若 partition key 選錯（例如全用同一個 tenant_id 下的單調遞增 ID），所有寫入會打到同一節點。解法：複合 partition key（((user_id, bucket)), message_id，bucket = hash % 16）、reverse timestamp、或 random salt（讀取時需 scatter-gather 多 partition）。

陷阱Secondary Index 濫用

Cassandra 的 secondary index 在高基數欄位上效能極差（需掃描所有節點）。正確做法是用查詢驅動的物化 table，或引入 Elasticsearch / Sai index 做搜尋側車。

一句話答法Wide-column 建模是「一個查詢一張表」的物化思維：用寫入放大與反正規化換取 partition-local 讀取，絕不指望 ad-hoc JOIN。
